# VQ-CAPITAL AI AGENT INSTRUCTION MANUAL

As an AI coding assistant working on any repository within the `VQ-Capital` organization, you MUST strictly adhere to the following architectural and coding constraints. Failure to do so violates the core principles of this High-Frequency Trading (HFT) platform.

## 1. TECHNOLOGY STACK CONSTRAINTS
- **Primary Language:** `Rust` (Strictly use `tokio` for async).
- **Secondary Language (for heavy AI/LLM math):** `C++`.
- **FORBIDDEN Languages for Data/Execution Path:** `Python`, `Node.js`, `JavaScript`. (Python is ONLY allowed for offline data analysis or ML model training, NEVER in production trading loops).
- **Messaging:** `NATS JetStream`. (FORBIDDEN: Kafka, RabbitMQ, Redis Pub/Sub).
- **Serialization:** `Protobuf` (prost in Rust). (FORBIDDEN: JSON for internal service communication).
- **Vector Database:** `Qdrant`.
- **Time-Series Database:** `QuestDB`.

## 2. CODING STANDARDS
- **Zero-Allocation:** Minimize memory allocations in hot loops. Use references and pre-allocated buffers.
- **Error Handling:** In Rust, ALWAYS use `Result` and `?`. Never use `.unwrap()` or `.expect()` in production logic unless gracefully handling a panic during startup.
- **Logging:** Use structured JSON logging (e.g., `tracing` crate in Rust) compatible with Grafana Loki.

## 3. ARCHITECTURAL BOUNDARIES
- **sentinel-ingest:** Only listens to external WebSockets and publishes Protobuf to NATS. NO database connections.
- **sentinel-storage:** Only listens to NATS and writes to Qdrant/QuestDB. NO trading logic.
- **sentinel-inference:** Reads from NATS, queries Qdrant, runs local C++ LLMs, publishes "Signal" events to NATS.
- **sentinel-execution:** Reads "Signal" events from NATS, manages risk limits, sends orders to Exchanges.
